<h2 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="highlights">Highlights<a class="hash-link" href="https://docusaurus.io/blog/releases/3.2#highlights" title="Direct link to Highlights">​</a></h2>
<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="faster-builds">Faster builds<a class="hash-link" href="https://docusaurus.io/blog/releases/3.2#faster-builds" title="Direct link to Faster builds">​</a></h3>
<p>We worked hard to reduce the time it takes to build a Docusaurus site in production mode.</p>
<p>Between v3.1.0 and v3.2.0, several changes have been made, leading to significantly faster production builds for many sites.</p>
<p>Let's take an example. Our benchmark on the <a class="" href="https://github.com/facebook/react-native-website/pull/4072" rel="noopener noreferrer" target="_blank">React Native website upgrading to v3.2</a> reports the following results:</p>
<ul>
<li class="">🔥 Cold builds: 95s ➡️ 66s (~30% faster)</li>
<li class="">🔥 Incremental builds: 55s ➡️ 22s (~60% faster)</li>
</ul>
<p>The results will vary depending on your site's topology and the options you turned on, but we expect the largest sites will see the most significant improvements.</p>
<p>Note that this is only the beginning, and Docusaurus performance can still be significantly improved, notably the bundling time and the memory consumption. Track our <a class="" href="https://github.com/facebook/docusaurus/issues/4765" rel="noopener noreferrer" target="_blank">performance issue</a> for upcoming improvements.</p>
<details class="details_Nokh alert alert--info details_Cn_P">What is the difference between a cold build and an incremental build?<div><div class="collapsibleContent_EoA1"><p>A cold build is when the Docusaurus caches are empty, generally after running <code>docusaurus clear</code>.</p><p>An incremental build happens when you run another time the <code>docusaurus build</code> command. Docusaurus automatically tries to "re-use" computations from former builds to make subsequent builds faster. In practice it's based on <a class="" href="https://webpack.js.org/guides/build-performance/#persistent-cache" rel="noopener noreferrer" target="_blank">Webpack persistent caching</a>. To enable incremental builds on your CI server, you can persist the <code>node_modules/.cache</code> folder across builds.</p></div></div></details>
<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="faster-dev-server">Faster Dev Server<a class="hash-link" href="https://docusaurus.io/blog/releases/3.2#faster-dev-server" title="Direct link to Faster Dev Server">​</a></h3>
<p>We also worked on improving the performance of the dev server, so that you can get a faster feedback when editing Markdown/MDX files.</p>
<p>The way we initially implemented content reloading wasn't great. For example, editing a blog post file would also trigger a useless reload of the unrelated docs plugin. From now on, when editing a plugin's content, only that plugin will reload. It's hard to measure precisely the impact of this change, but I estimate edits should appear in your browser at least 50% faster 🔥.</p>
<p>We plan to keep improving the speed of our dev server, with even more granular hot reloads, ensuring we don't run useless computations that would always keep giving the same result.</p>
<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="mdx-partials-table-of-contents">MDX partials table-of-contents<a class="hash-link" href="https://docusaurus.io/blog/releases/3.2#mdx-partials-table-of-contents" title="Direct link to MDX partials table-of-contents">​</a></h3>
<p>With <a class="" href="https://github.com/facebook/docusaurus/pull/9684" rel="noopener noreferrer" target="_blank">#9684</a>, Docusaurus is now able to render headings coming from an imported partial into the table-of-contents.</p>
<p>Docusaurus and MDX allows you to <a class="" href="https://docusaurus.io/docs/markdown-features/react#importing-markdown">import one Markdown file into another</a>. We usually call the imported Markdown file a "partial", and use the <code>_</code> prefix so that this file does not lead to the creation of a new page.</p>

<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="blog-improvements">Blog improvements<a class="hash-link" href="https://docusaurus.io/blog/releases/3.2#blog-improvements" title="Direct link to Blog improvements">​</a></h3>
<p>We improved the blog plugin with several new options to make it even more powerful and flexible:</p>
<ul>
<li class=""><a class="" href="https://github.com/facebook/docusaurus/pull/9912" rel="noopener noreferrer" target="_blank">#9912</a>: you can now display the last update time and author of a blog post, a feature the docs plugin already had.</li>
<li class=""><a class="" href="https://github.com/facebook/docusaurus/pull/9886" rel="noopener noreferrer" target="_blank">#9886</a>: a new <code>processBlogPosts</code> option allow you to filter/transform/sort blog posts.</li>
<li class=""><a class="" href="https://github.com/facebook/docusaurus/pull/9838" rel="noopener noreferrer" target="_blank">#9838</a>: a new <code>pageBasePath</code> option allows you to customize the blog pagination URL segment (<code>/blog/page/2</code>)</li>
</ul>
<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="sitemap-lastmod">Sitemap lastmod<a class="hash-link" href="https://docusaurus.io/blog/releases/3.2#sitemap-lastmod" title="Direct link to Sitemap lastmod">​</a></h3>
<p>With <a class="" href="https://github.com/facebook/docusaurus/pull/9954" rel="noopener noreferrer" target="_blank">#9954</a>, the sitemap plugin has a new <code>lastmod</code> option that can now emit a <code>&lt;lastmod&gt;</code> tag on the XML. The value is read from the Git history by default, but can be overridden with docs and blog <code>last_update</code> front matter.</p>
<p>We also made it possible to opt-out of emitting <code>&lt;priority&gt;</code> and <code>&lt;frequency&gt;</code> tags, which are generally ignored by crawlers (notably <a class="" href="https://developers.google.com/search/blog/2023/06/sitemaps-lastmod-ping" rel="noopener noreferrer" target="_blank">Google</a>).</p>
