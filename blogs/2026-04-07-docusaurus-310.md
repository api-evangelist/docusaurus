<p>We are happy to announce <strong>Docusaurus 3.10</strong>.</p>
<p>This will be the <strong>last release in the v3.x line</strong>, and it helps you <strong>prepare for Docusaurus v4</strong>.</p>
<p>Upgrading is easy. We follow <a class="" href="https://semver.org/" rel="noopener noreferrer" target="_blank">Semantic Versioning</a>, and minor version updates have <strong>no breaking changes</strong>, according to our <a class="" href="https://docusaurus.io/community/release-process">release process</a>.</p>
<p>However, if you opt in for upcoming Docusaurus v4 breaking changes globally with <code>future.v4: true</code>, make sure to review the dedicated section below.</p>

<h2 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="v4-future-flags">v4 Future Flags<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#v4-future-flags" title="Direct link to v4 Future Flags">​</a></h2>
<p>The <a class="" href="https://docusaurus.io/docs/api/docusaurus-config#future">Docusaurus v4 Future Flags</a> let you <strong>opt in for upcoming Docusaurus v4 breaking changes</strong> incrementally and prepare for Docusaurus v4 ahead of time. You can enable them all at once with <code>future.v4: true</code>.</p>
<p>This release introduces new flags and breaking changes that might break your site if you use <code>future.v4: true</code> and upgrade:</p>
<ul>
<li class=""><a class="" href="https://github.com/facebook/docusaurus/pull/11797" rel="noopener noreferrer" target="_blank"><code>future.v4.siteStorageNamespacing</code></a>: In Docusaurus v4, <code>localStorage</code> keys will be automatically namespaced by default (<code>theme</code> =&gt; <code>theme-&lt;hash&gt;</code>) to prevent key conflicts. This is likely to reset your site's visitor storage unless you migrate it to new key names. See the dedicated <a class="" href="https://docusaurus.io/blog/releases/3.10#site-storage">Site Storage section</a> below.</li>
<li class=""><a class="" href="https://github.com/facebook/docusaurus/pull/11802" rel="noopener noreferrer" target="_blank"><code>future.v4.fasterByDefault</code></a>: Docusaurus Faster is now stable and will be enabled by default in Docusaurus v4. See the dedicated <a class="" href="https://docusaurus.io/blog/releases/3.10#docusaurus-faster">Docusaurus Faster section</a> below.</li>
<li class=""><a class="" href="https://github.com/facebook/docusaurus/pull/11896" rel="noopener noreferrer" target="_blank"><code>future.v4.mdx1CompatDisabledByDefault</code></a>: In Docusaurus v4, MDX v1 compatibility options will be disabled by default, and you might have to adjust your docs accordingly. See the dedicated <a class="" href="https://docusaurus.io/blog/releases/3.10#strict-mdx">Strict MDX section</a> below.</li>
</ul>
<h2 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="security">Security<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#security" title="Direct link to Security">​</a></h2>
<p><strong><a class="" href="https://github.blog/security/supply-chain-security/our-plan-for-a-more-secure-npm-supply-chain/" rel="noopener noreferrer" target="_blank">npm supply chain attacks</a></strong> have surged recently. A single compromised maintainer or package can ripple through the ecosystem and affect thousands of downstream users, as seen with the <a class="" href="https://socket.dev/blog/hidden-blast-radius-of-the-axios-compromise" rel="noopener noreferrer" target="_blank">axios compromise</a>.</p>
<p>We’ve taken steps to strengthen our supply chain, and recommend securing your own site with additional measures as well.</p>
<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="trusted-publishing">Trusted Publishing<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#trusted-publishing" title="Direct link to Trusted Publishing">​</a></h3>
<p>We adopted <strong><a class="" href="https://docs.npmjs.com/trusted-publishers" rel="noopener noreferrer" target="_blank">npm Trusted Publishing</a></strong> for <a class="" href="https://github.com/facebook/docusaurus/pull/11819" rel="noopener noreferrer" target="_blank">stable</a> and <a class="" href="https://github.com/facebook/docusaurus/pull/11712" rel="noopener noreferrer" target="_blank">canary</a> releases.</p>
<p>Releases are now published through a single <code>publish.yml</code> GitHub Actions workflow using short-lived OIDC tokens. Each release has verifiable provenance, with a transparency log showing how and when it was published.</p>

<p><img alt="npm trusted publishing provenance details" class="img_vXGZ" height="696" src="https://docusaurus.io/assets/images/provenance-4b7dc023c1c4709717b5100e9db1d566.jpg" width="2860" /></p>
<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="security-workflow">Security Workflow<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#security-workflow" title="Direct link to Security Workflow">​</a></h3>
<p>In <a class="" href="https://github.com/facebook/docusaurus/pull/11874" rel="noopener noreferrer" target="_blank">#11874</a>, we introduced a new security workflow that runs daily and on every pull request. It scans for suspicious dependency updates affecting our official packages and their transitive dependencies:</p>
<ul>
<li class="">Install the Docusaurus site template through the <a class="" href="https://socket.dev/blog/introducing-socket-firewall" rel="noopener noreferrer" target="_blank">Socket Firewall</a> to detect known malware in our dependency graph</li>
<li class="">Check for unexpected <code>preinstall</code> and <code>postinstall</code> lifecycle scripts, using <a class="" href="https://pnpm.io/settings#strictdepbuilds" rel="noopener noreferrer" target="_blank">pnpm <code>strictDepBuilds</code></a></li>
<li class="">Check for GitHub repository and tarball URL dependencies, using <a class="" href="https://pnpm.io/settings#blockexoticsubdeps" rel="noopener noreferrer" target="_blank">pnpm <code>blockExoticSubdeps</code></a></li>
<li class="">Check for packages that downgrade their trust level, using <a class="" href="https://pnpm.io/settings#trustpolicy" rel="noopener noreferrer" target="_blank">pnpm <code>trustPolicy: no-downgrade</code></a></li>
<li class="">Run similar checks on our own monorepo, to protect Docusaurus maintainers and contributors</li>
</ul>

It is designed to detect serious vulnerabilities affecting the Docusaurus ecosystem as early as possible, so we can react quickly and notify you.</p><p>With semver dependency ranges, it's impossible to guarantee a 100% secure supply chain, since any new npm release in our dependency graph can introduce a vulnerability. Ultimately, securing your site is your responsibility. At a minimum, rely on a lockfile and upgrade dependencies deliberately and cautiously.</p></div></div>

<h2 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="strict-mdx">Strict MDX<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#strict-mdx" title="Direct link to Strict MDX">​</a></h2>
<p>This release introduces new MDX options to encourage stricter usage of native MDX syntax, instead of relying on proprietary Docusaurus syntax on top of MDX.</p>
<p>Historically, Docusaurus compiled your files with MDX v1, which was quite forgiving. Since then, the ecosystem has widely moved to MDX v3, which is stricter. Docusaurus v3.0 introduced <code>markdown.mdx1Compat</code> to help you upgrade incrementally.</p>
<p>In Docusaurus v4, we plan to turn the <code>markdown.mdx1Compat</code> options off by default. This upcoming change is now part of our <a class="" href="https://docusaurus.io/blog/releases/3.10#v4-future-flags">v4 future flags</a> (<code>future.v4.mdx1CompatDisabledByDefault: true</code>).</p>
<p>We'd like the community to adopt a stricter, native MDX syntax for a few reasons:</p>
<ul>
<li class="">This makes your docs more portable: external tools like Prettier, ESLint, TypeScript, VS Code, and GitHub can understand them.</li>
<li class="">Docusaurus doesn't need to pre-process your documents with regular expressions before MDX compilation.</li>
<li class="">This improves compatibility with the <a class="" href="https://unifiedjs.com/" rel="noopener noreferrer" target="_blank">Unified</a> ecosystem (Remark, MDX) and the <a class="" href="https://mdxjs.com/playground/" rel="noopener noreferrer" target="_blank">MDX Playground</a>.</li>
</ul>
<h3 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="strict-extensions">Strict Extensions<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#strict-extensions" title="Direct link to Strict Extensions">​</a></h3>
<p>We believe that:</p>
<ul>
<li class=""><code>.md</code> should be parsed as CommonMark</li>
<li class=""><code>.mdx</code> should be parsed as MDX</li>
</ul>
<p>In reality, Docusaurus has only ever supported MDX, and we should have used the <code>.mdx</code> extension from the start. Newly initialized sites now use <code>.mdx</code> by default (<a class="" href="https://github.com/facebook/docusaurus/pull/11897" rel="noopener noreferrer" target="_blank">#11897</a>). We encourage you to rename your existing files to <code>.mdx</code> as well, so that external tools can understand your content is MDX.</p>
<div class="theme-admonition theme-admonition-info admonition_o5H7 alert alert--info"><div class="admonitionHeading_FzoX"><span class="admonitionIcon_rXq6"><svg viewBox="0 0 14 16" xmlns="http://www.w3.org/2000/svg"><path d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z" fill-rule="evenodd"></path></svg></span>About CommonMark support</div><div class="admonitionContent_Knsx"><p>Although we also have experimental support for CommonMark, it still doesn't have full feature parity with MDX (<a class="" href="https://github.com/facebook/docusaurus/issues/9092" rel="noopener noreferrer" target="_blank">issue</a>). Once we reach full feature parity, we'll default to <code>markdown.format: 'detect'</code> to ensure that <code>.md</code> files are parsed as CommonMark instead of MDX.</p></div></div>

<h2 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="vcs">VCS API - Experimental<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#vcs" title="Direct link to VCS API - Experimental">​</a></h2>

<p>We have also noticed that our historical strategy to read from the Git history can be a <strong>significant build performance bottleneck</strong> for large Docusaurus sites. Issuing one <code>git log &lt;filepath&gt;</code> call per MDX file doesn't scale well.</p>
<p>To solve this bottleneck, we implemented a strategy that reads the whole Git repository eagerly in a single <code>git log</code> call, leading to significant performance improvements. This strategy is now part of <a class="" href="https://docusaurus.io/blog/releases/3.10#docusaurus-faster">Docusaurus Faster</a> (<code>future.faster.gitEagerVcs: true</code>) and will become the default in Docusaurus v4.</p>
<p>We also provide built-in VCS preset strategies. The available presets are:</p>
<ul>
<li class=""><code>git-ad-hoc</code>: the historical strategy based on <code>git log &lt;filename&gt;</code> calls</li>
<li class=""><code>git-eager</code>: the new Git strategy that reads your whole repository upfront</li>
<li class=""><code>hardcoded</code>: returns a hardcoded value, useful in dev/tests to speed up DX</li>
<li class=""><code>disabled</code>: returns null for all files, considering them untracked</li>
<li class=""><code>default-v1</code>: the historical default (dynamic, <code>git-ad-hoc</code> in prod, <code>hardcoded</code> in dev)</li>
<li class=""><code>default-v2</code>: the upcoming Docusaurus v4 default (dynamic, <code>git-eager</code> in prod, <code>hardcoded</code> in dev)</li>
</ul>

<h2 class="anchor anchorTargetHideOnScrollNavbar_qJOx" id="other-changes">Other changes<a class="hash-link" href="https://docusaurus.io/blog/releases/3.10#other-changes" title="Direct link to Other changes">​</a></h2>
<p>Other notable changes include:</p>
<ul>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11843" rel="noopener noreferrer" target="_blank">#11843</a>, we now use TypeScript 6.0 for newly initialized sites. However, it requires <code>"ignoreDeprecations": "6.0"</code> for now.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11571" rel="noopener noreferrer" target="_blank">#11571</a>, the <code>siteConfig.headTags</code> API now accepts custom HTML elements.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11675" rel="noopener noreferrer" target="_blank">#11675</a>, the live code block theme now has a button to reset the playground.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11734" rel="noopener noreferrer" target="_blank">#11734</a>, we split the <code>&lt;DocCard&gt;</code> component to make it easier to extend/swizzle. It's now easier to assign custom emojis for the docs generated index page.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11733" rel="noopener noreferrer" target="_blank">#11733</a>, the <code>&lt;Tabs&gt;</code> component now uses React context instead of props, making it possible to create custom <code>&lt;TabItem&gt;</code> components.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11696" rel="noopener noreferrer" target="_blank">#11696</a>, all newly initialized TypeScript sites will have <code>"strict: true"</code> by default.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11611" rel="noopener noreferrer" target="_blank">#11611</a>, we made it possible to create a new Docusaurus site in <code>.</code>, the current directory.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11666" rel="noopener noreferrer" target="_blank">#11666</a>, the pages plugin can now use Markdown file path links (<code>[text](./other-page.md)</code>), as the docs and blog plugins already support it.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11642" rel="noopener noreferrer" target="_blank">#11642</a>, admonitions now support class/id shortcuts, such as <code>:::note{.my-class #my-id}</code>.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11541" rel="noopener noreferrer" target="_blank">#11541</a> and <a class="" href="https://github.com/facebook/docusaurus/pull/11683" rel="noopener noreferrer" target="_blank">#11683</a>, we made sure Docusaurus is compatible with the latest version of Algolia DocSearch 4.x, unlocking new features such as AskAI Suggested Questions.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11684" rel="noopener noreferrer" target="_blank">#11684</a> and <a class="" href="https://github.com/facebook/docusaurus/pull/11653" rel="noopener noreferrer" target="_blank">#11653</a>, we removed many third-party dependencies from our <code>create-docusaurus</code> CLI, making it faster to create a new site.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11794" rel="noopener noreferrer" target="_blank">#11794</a>, we fixed a long-standing bug that prevented the translation of category index page titles in pagination links.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11784" rel="noopener noreferrer" target="_blank">#11784</a>, we changed the syntax we recommend for math equations, preferring regular Markdown code blocks over <code>$$</code> to improve docs portability.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11753" rel="noopener noreferrer" target="_blank">#11753</a>, we added a basic <code>AGENTS.md</code> file. As a reminder, any AI usage in Docusaurus contributions must be disclosed.</li>
<li class="">In <a class="" href="https://github.com/facebook/docusaurus/pull/11698" rel="noopener noreferrer" target="_blank">#11698</a>, we upgraded our monorepo to React 19. We'll drop support for React 18 in Docusaurus v4.</li>
</ul>
